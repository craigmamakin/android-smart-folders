# Pull Request & Branching Guidelines

To maintain a clean and professional codebase for **Smart Folders**, please follow these guidelines when contributing.

## 1. Branching Strategy
We use a structured branching naming convention. Every branch must be linked to an issue.

### Branch Name Format
`issue_[ISSUE-NUMBER]_[brief-descriptive-title]`

*   **Example**: `issue_102_fix_widget_labels`
*   **Max Length**: Please keep branch names under 50 characters.
*   **Lowercase**: Use lowercase letters and underscores only.

## 2. Pull Request (PR) Requirements
All changes must be merged into the `main` branch via a Pull Request.

### PR Process:
1.  **Sync**: Ensure your local `main` branch is up to date before creating a feature branch.
2.  **Raise PR**: Target the `main` branch.
3.  **Title**: The PR title should be clear and concise (e.g., "Refactor: Modularize AppManager logic").
4.  **Description**:
    *   Link the issue number (e.g., "Closes #102").
    *   Provide a brief summary of what was changed.
    *   Include a screenshot if there are UI/Visual changes.

## 3. Commit Messages
*   Keep them imperative: "Add high-contrast shadow" instead of "Added high-contrast shadow".
*   Reference the issue number in the first commit of the branch.

---
*Following these guidelines ensures a seamless integration process and a "Pixel-perfect" project history.*
